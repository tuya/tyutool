# Batch Auth 批量授权使用指南

本文说明 tyuTool 图形界面中 **Batch Auth** 标签页的能力：通过串口向设备写入 TuyaOpen 授权信息，并维护授权 Excel（`.xlsx`）。

- **法律与条款**：使用前请阅读仓库根目录下的 [Terms_And_Agreements.md](../../Terms_And_Agreements.md)。
- **英文版**：[Batch-Auth-Guide.md](../en/Batch-Auth-Guide.md)。

---

## 使用前必读

在开始授权或选择授权表进行批量任务前，请确认：

| 事项 | 说明 |
| --- | --- |
| **备份授权码清单 Excel 表** | 在点击「开始授权」或加载表进行批量任务前，**自行将 `.xlsx` 完整备份**到安全位置（独立目录、版本管理或团队网盘等）。授权成功后会**直接改写**表内行（状态、MAC、时间戳等）。程序可能在同目录生成 `原文件名.xlsx.bak`，或在首次确认时尝试复制备份，但**不能替代**你的主动备份。产线开始前请再次确认备份可恢复。 |
| **MAC 唯一** | 工具按设备上报的 **MAC** 与 Excel 记录关联；**不会校验**不同模组之间 MAC 是否重复。须在产线与物料层面保证每台模组 MAC 在业务范围内唯一，否则可能导致授权记录错乱。 |
| **固件与命令** | 设备须启用 TAL CLI，并注册 **`auth`**、**`auth-read`**、**`read_mac`**（详见下文「固件与协议要求」）。 |

---

## 功能概览

- **主界面**：**Batch Auth** 页与后台工作线程（`AuthWorker`）配合；点击 **Start** 后，授权与烧录在子线程执行，不阻塞界面。
- **核心模块**：**AuthHandler** 负责串口、与设备交互、读写授权并驱动 Excel；**AuthExcelParser** 负责解析与保存 `.xlsx`。
- **可选先烧录**：若同时指定芯片与固件文件，线程内先完成烧录与复位，再进入与「仅授权」相同的串口授权流程。

## 操作步骤

### 1. 环境准备

1. 运行 **tyuTool GUI**（`tyutool_gui.py` 或已打包的可执行文件）。
2. USB 转串口已连接设备且驱动正常（Linux 上常见为 `/dev/ttyUSB*`、`/dev/ttyACM*` 等）。
3. 准备好业务侧提供的 **授权表 `.xlsx`**，并完成上表「备份授权码清单 Excel 表」要求。
4. Excel 处理依赖 **openpyxl**（见项目 `requirements.txt`）。

### 2. 授权表格式

- **第一行为表头**，列名不区分大小写，支持别名：
  - **必填**：`UUID`；`AUTHKEY` 或 **`key`**（二者都表示授权密钥列）。
  - **可选**：若已有 `STATUS`、`MAC`、`TIMESTAMP` 列则使用；若无，程序在首次需要写入时会**自动追加**。
- **统计**：自第 2 行起，**UUID 非空**计为一条；`STATUS` 为 `USED`（不区分大小写）计为已使用。
- **分配**：自上而下取**第一条** UUID、AUTHKEY 均有值且**未标记 USED** 的记录作为本次写入的授权。

### 3. 界面操作

**串口与波特率**

- **Port**：点击 **Rescan** 后选择设备串口。
- **Baud**：ESP32 为 `115200`。

**选择授权表**

- 通过 **Browse** 选择 `.xlsx`；加载成功后显示 **Total / Used / Remain**。
- 界面会提示将**修改 Excel**；请再次确认已自行备份。确认后若同目录尚无 `.bak`，程序可能生成 **`原文件名.xlsx.bak`**。

**（可选）先烧录再授权**

- **Chip** 选 **ESP32**，**Firmware** 选择 `.bin`。
- 不选固件时，不包含烧录与 DTR/RTS 复位，仅串口授权。

**开始授权**

- 已选 **Port** 且已选 **Excel** 后，**Start** 可用；运行中显示步骤列表与进度。

**仅读 MAC**

- 选好 **Port** 与 **Baud** 后点击 **Read MAC**：只读 MAC，不写授权、不改表。

### 4. 会话日志

- 每次授权会话在**授权 Excel 所在目录**生成独立日志，文件名形如：`表名_auth_时间戳.log`；界面可复制完整路径。
- 建议每次操作结束后将当次及相关日志**另行归档**（按日期或批次），便于追溯与排障。

## 固件与协议要求

### 串口 CLI：auth、auth-read、read_mac

Batch Auth 通过**串口**与设备上的 **TAL CLI** 交互，**依赖固件侧已注册且可用的三条命令**：**`auth`**、**`auth-read`**、**`read_mac`**。名称与交互语义须与 tyuTool 内置协议一致；若未启用或未实现等效命令，**开始授权**与 **Read MAC** 无法完成。

在 **TuyaOpen** 公开仓库中，可按以下路径对照集成（路径相对于该仓库根目录）：

- **`apps/tuya_cloud/switch_demo/src/tuya_main.c`**：在 `user_main` 中，`tal_kv_init` 之后依次调用 **`tal_cli_init`**、**`tuya_authorize_init`**、**`tuya_app_cli_init`**，用于启动 CLI、注册授权子命令以及应用层命令表。若目标为 Linux 桌面等非嵌入式环境，部分初始化可能被条件编译跳过，以实际工程为准。
- **`src/tuya_cloud_service/authorize/tuya_authorize.c`**：**`tuya_authorize_init`** 向 CLI 注册 **`auth`**（按固定长度将 UUID、AuthKey 写入 KV）、**`auth-read`**（从 KV 读出当前授权）、**`auth-reset`**（按需清除 KV 授权项）等；KV 键名例如 **`UUID_TUYAOPEN`**、**`AUTHKEY_TUYAOPEN`**。Batch Auth 主要依赖 **`auth`** 与 **`auth-read`** 与 Excel/设备状态对齐。
- **`apps/tuya_cloud/switch_demo/src/cli_cmd.c`**：**`tuya_app_cli_init`** 将 **`read_mac`** 等命令注册到 **`s_cli_cmd`**。参考实现里 **`read_mac`** 常与 Wi‑Fi 相关宏一起编译；若芯片或网络方案不同，**仍须提供与协议兼容的 `read_mac`**（例如从本机网络栈读取 MAC 并输出），保证 PC 端解析到的 MAC 格式符合预期。

集成时请将上述调用与命令注册纳入应用启动流程；仅烧录固件而未打开对应 CLI 与三条命令时，Batch Auth 仍会失败。

### 限时关闭 CLI（须自行在固件中实现）

若希望在一段时间后使上述 CLI 命令（或整条串口调试/授权通道）失效，**须在固件中自行实现**，例如：**上电计时或累计运行时间到达阈值后不再注册相关命令**，或**发布镜像中不调用 `tal_cli_init` / 授权模块初始化**。**tyuTool 不提供「到期自动关闭 CLI」能力**。

### KV 密钥与数据安全（防泄露）

若不希望模组 **KV 存储**（含授权相关信息）因使用公开、默认或与竞品相同的密钥而被他人按同样参数解密读取，应在**设备固件**中自行修改 **`tal_kv_init`** 配置：将 **`seed`**、**`key`** 替换为**仅你方掌握**的随机串，并妥善保管、切勿泄露。KV 数据通常依赖该 **`seed` / `key`** 参与派生或加解密；密钥一旦泄露，对应 KV 内敏感内容（含授权信息）将面临被还原的风险。

示例（仅示意字段结构，**须改为你方自行生成且保密的值**，不得长期沿用文档、示例或 SDK 公开默认值）：

```c
tal_kv_init(&(tal_kv_cfg_t){
    .seed = "vmlkasdh93dlvlcy",
    .key  = "dflfuap134ddlduq",
});
```

生产与发布流程中需防止密钥进入公开仓库、日志或外发文档。

## 工作原理（实现摘要）

本节说明工具如何串联界面、后台线程、串口协议与 Excel，便于理解现象与日志；不要求终端用户阅读源码。

### 整体与 Excel

- 启动授权时加载工作簿，解析表头中的 **UUID**、**AUTHKEY**（或 **key**）等列；若缺少 **STATUS / MAC / TIMESTAMP**，在需要写入磁盘时自动追加。
- 对同一文件使用**文件锁**（`.lock`），避免多实例同时改写；统计 **总数 / 已用 / 剩余** 未用行。
- 写入设备并校验成功后，将对应行标记为 **USED**，并写入 **MAC** 与**时间戳**，然后保存；首次保存前可能对原表做 **`.bak` 备份**（若尚不存在）。

### 串口与可选烧录

- **未选择固件**：打开串口 → 排空上电启动阶段串口数据以便进入可交互状态 → 读 MAC 与授权。
- **已选择固件（当前为 ESP32）**：先校验串口 → 按芯片调用既有**烧录流程**（握手、擦除、写入、校验、重启）→ 通过 **DTR/RTS** 复位 → 再按授权波特率打开串口，进入读 MAC、授权与校验。

### 单台设备授权逻辑（摘要）

流程默认**每台物理设备有唯一 MAC**；若多台模组 MAC 相同，按 MAC 匹配与写入的逻辑可能无法区分真实设备。

1. **读 MAC**（失败会按上限重试）。
2. **读设备当前授权**：若设备为占位或未授权，则按表分配新行；若已授权，则与 Excel 中**按 MAC 匹配的行**比对。
3. **决策**：若与表中记录一致可跳过写入；若不一致或 MAC 不在表中，可能通过对话框询问是否**覆盖**（用户可 **Copy** 提示内容）。
4. **写入与校验**：从表中取下一条未使用的 UUID/AUTHKEY（或复用某行），经协议**写入**设备，再**回读比对**；全部通过后才**回写 Excel** 标记已用。

### 界面步骤条

界面步骤（打开串口、烧录、复位、读 MAC、写入授权、校验、关闭串口等）与上述阶段对应，用于展示进度与失败位置。

## 常见问题

| 现象 | 处理建议 |
| --- | --- |
| **Remain 为 0** | 无可用行；补充授权或检查是否均已 USED。 |
| **读 MAC 失败** | 检查接线、波特率、固件是否进入可交互状态；确认已启用 CLI 且 **`read_mac`** 可用；可先烧录支持授权协议的固件。 |
| **Excel 报错** | 确认首行包含 **UUID** 与 **AUTHKEY**（或 **key**）。 |
| **无法加锁 / 文件被占用** | 关闭其他正在使用该 `.xlsx` 的 tyuTool 实例；同一表使用 `.lock`，多开可能导致后者无法加锁。 |

**其他约定**

- **备份**：以自行备份为准；程序 `.bak` 仅为辅助。
- **MAC**：须自行保证各模组 MAC 在业务范围内唯一；工具不检测跨设备 MAC 冲突。
- **烧录**：芯片下拉当前为 **ESP32**；未选固件则不执行 Flash。
